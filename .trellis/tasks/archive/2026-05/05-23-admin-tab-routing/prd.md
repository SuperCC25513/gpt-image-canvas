# 后台菜单路由化

## 目标

把后台管理的一级菜单从组件内部 tab 状态改成 URL 可表达的路由状态，让用户、开发者、测试和审阅反馈都可以直接定位到具体后台模块。

## 用户价值

- 后台模块可以直接分享和复现，例如生成服务配置可以打开到固定 URL。
- 刷新页面不会丢失当前后台菜单位置。
- 浏览器前进/后退能按后台菜单切换记录工作。
- 后续后台模块扩展时，路由、数据加载和权限边界更清晰。

## 已确认事实

- 当前应用顶层路由在 `apps/web/src/features/canvas/CanvasApp.tsx` 中由 `routeFromLocation()`、`pathForRoute()` 和 `navigateToRoute()` 管理。
- 当前顶层后台路由只有 `/admin`，对应 `AppRoute = "admin"`。
- 当前后台菜单在 `apps/web/src/features/admin/AdminPage.tsx` 中由 `useState<AdminTab>("users")` 管理。
- 当前后台一级菜单包括：用户管理、生成服务、生成审计、系统设置。
- 当前后台菜单切换不会改变 URL，刷新 `/admin` 会回到默认“用户管理”。
- 生成服务配置已经迁移到后台“生成服务”页，适合作为可直达路由。

## 需求

- 后台一级菜单需要进入 URL 路由管理。
- 推荐路径：
  - `/admin/users`：用户管理。
  - `/admin/providers`：生成服务。
  - `/admin/audits`：生成审计。
  - `/admin/settings`：系统设置。
- `/admin` 需要兼容旧入口，并默认进入用户管理。
- 点击后台一级菜单时应更新浏览器地址和历史记录。
- 直接打开后台子路径时，应选中对应菜单并展示对应内容。
- 刷新后台子路径后，应保持在同一个后台菜单。
- 浏览器返回/前进应能在后台菜单之间切换。
- 未知后台子路径需要有明确兼容策略，推荐回退到 `/admin/users`。
- 顶部导航“后台”仍作为后台入口，默认进入用户管理。
- 现有后台鉴权和 API 权限校验不因前端路由调整而放松。
- 新增用户可见文案需同步 `zh-CN` 和 `en`，如有新增文案。

## 验收标准

- [ ] 打开 `/admin` 后进入默认后台页面，并且行为向后兼容。
- [ ] 打开 `/admin/users` 时显示“用户管理”并选中对应菜单。
- [ ] 打开 `/admin/providers` 时直接显示生成服务配置，不需要额外点击后台菜单。
- [ ] 打开 `/admin/audits` 时显示“生成审计”并选中对应菜单。
- [ ] 打开 `/admin/settings` 时显示“系统设置”并选中对应菜单。
- [ ] 在后台菜单之间点击切换时，浏览器 URL 同步变化。
- [ ] 在后台菜单之间切换后，浏览器返回/前进能恢复前后菜单状态。
- [ ] 刷新任一后台子路径后仍停留在对应菜单。
- [ ] 未知后台子路径不会空白或展示错误模块。
- [ ] `pnpm typecheck` 和 `pnpm build` 通过。
- [ ] 使用内置 Browser 验证桌面和移动视口的后台子路由。

## 暂不包含

- 不重做后台权限模型。
- 不拆分后台为多个独立页面包或服务端路由。
- 不改变后台 API 字段语义。
- 不改变生成服务配置本身的加载、保存和密钥掩码逻辑。

## 已确认决策

- 采用子路径方案 `/admin/users`、`/admin/providers`、`/admin/audits`、`/admin/settings`，不使用 query/hash 方案。
