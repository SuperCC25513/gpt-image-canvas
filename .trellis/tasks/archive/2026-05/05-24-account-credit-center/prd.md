# 用户名菜单与积分中心

## 目标

把当前顶部直接暴露的积分和签到入口收进用户名下拉菜单，并新增「积分中心」页面，让用户能集中查看当前余额、每日签到状态和积分流水。

当前顶部直接展示「积分 + 签到」占用导航空间，也不适合作为长期账户能力入口。用户期望从用户名进入账户相关操作，包括退出和积分中心。

## 已确认事实

- 后端已经有积分余额字段：`users.credits`。
- 后端已经有积分流水表：`credit_transactions`。
- 已记录的流水原因包括：
  - `registration_bonus`：注册赠送。
  - `daily_checkin`：每日签到。
  - `generation_charge`：生成扣费。
  - `generation_refund`：生成失败退款。
  - `admin_adjustment`：后台调整。
- 后端已有账户状态接口：`GET /api/auth/me`，返回当前用户、积分设置和今日签到状态。
- 后端已有签到接口：`POST /api/checkin`。
- 当前没有发现用户侧积分流水查询接口。
- 当前顶部账号区已经展示用户名、积分、签到和退出按钮，但用户不满意这种直接展示方式。

## 需求

- 顶部账号区域改为用户名按钮或账号入口。
- 点击用户名后打开下拉菜单。
- 下拉菜单至少包含：
  - 当前用户信息。
  - 积分余额摘要。
  - 入口「积分中心」。
  - 操作「退出登录」。
- 顶部导航不再直接常驻展示完整积分和签到按钮。
- 新增独立路由积分中心页面 `/credits`，用于展示：
  - 当前积分余额。
  - 今日签到状态和签到入口。
  - 积分流水列表。
- 新增用户侧积分流水查询接口，只返回当前登录用户自己的流水。
- 积分流水应展示积分增减、类型、时间和关联信息。
- 签到成功后应刷新余额、签到状态和流水列表。
- 积分中心页面加载失败时应有可理解错误反馈。
- 页面和菜单需要支持桌面与移动端，不挤压顶部导航。

## 验收标准

- [ ] 顶部账号区域只保留用户名入口，不再把完整积分和签到按钮直接铺在导航栏。
- [ ] 点击用户名能打开下拉菜单。
- [ ] 下拉菜单可进入积分中心。
- [ ] 积分中心使用独立路由 `/credits`。
- [ ] 下拉菜单可退出登录。
- [ ] 积分中心能显示当前积分余额。
- [ ] 积分中心能显示今日签到状态，未签到时可签到。
- [ ] 积分中心能显示积分流水列表。
- [ ] 生成扣费、签到、退款、后台调整等流水类型有清晰文案。
- [ ] 签到成功后余额、签到状态、流水同步更新。
- [ ] 首页、画布、提示池、作品库、简单生成页顶部导航在桌面和移动端都不发生明显挤压。
- [ ] `pnpm typecheck` 和 `pnpm build` 通过。
- [ ] 浏览器验证主要路径：打开菜单、进入积分中心、签到、返回业务页面。

## 非目标

- 不新增新的积分规则。
- 不改变生成扣费、退款、签到、后台调整的后端业务规则。
- 不做充值、支付或积分兑换。
- 不做管理员全局流水审计页面。
- 不改现有后台积分设置能力。

## 已确认方案

- 积分中心采用独立路由页面 `/credits`，不做弹窗。
- 用户名下拉菜单提供进入 `/credits` 的入口。

选择独立页面的原因：积分流水列表需要分页、空状态、错误状态和后续扩展空间，独立页面比菜单弹窗更稳定。
