# 管理后台生成服务配置平铺

## 目标

把管理后台“生成服务”配置从二级弹窗主路径改成后台页面内平铺展示，让管理员进入对应标签页后可以直接查看和编辑图像模型、Agent 大模型、来源顺序、来源详情与 Codex fallback。

## 用户价值

- 管理员减少一次“打开生成服务配置”的额外点击。
- 生成服务可用性、当前来源和配置项更容易被发现。
- 保留现有配置逻辑，降低保存、刷新、密钥掩码和错误提示分叉风险。

## 已确认事实

- 当前生成服务配置主要由 `ProviderConfigDialog` 承载。
- 管理后台已有“生成服务”相关入口。
- 生成服务配置包含图像模型、Agent 大模型、来源顺序、来源详情、保存、刷新、Codex 登录/退出等动作。

## 需求

- 管理后台“生成服务”标签页打开后，应直接平铺展示生成服务配置内容。
- 主路径不应只显示一个“打开生成服务配置”的按钮。
- 平铺页面应复用现有加载、保存、刷新、密钥掩码、错误提示和 Codex 登录/退出逻辑。
- 现有弹窗组件可以保留给其他入口复用，但后台主路径必须直接显示同一份配置数据。
- 新增或调整用户可见文案需同步 `zh-CN` 和 `en`。

## 验收标准

- [ ] 管理后台“生成服务”标签页打开后无需额外点击弹窗按钮，即可查看图像模型、Agent 大模型、来源顺序和来源详情。
- [ ] 管理员可在平铺页面保存配置并看到现有成功/失败反馈。
- [ ] 平铺配置与原配置弹窗复用同一套数据加载、保存、刷新、密钥掩码和错误提示逻辑。
- [ ] Codex 登录/退出与可用性状态在平铺页面可用。
- [ ] 原有非后台入口如仍使用弹窗，不被破坏。
- [ ] 桌面和移动视口无明显溢出或遮挡。
- [ ] `pnpm typecheck` 和 `pnpm build` 通过。

## 暂不包含

- 不新增 Provider 类型或配置字段。
- 不改变 Provider 来源优先级语义。
- 不调整简单生图模式或画布生成流程。

## 后续规划要求

- 该任务涉及后台 UI 和 Provider 配置复用；实现前需要补充 `design.md` 和 `implement.md`，并单独评审。
